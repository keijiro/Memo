# glTFast

.glb ファイルはテクスチャ埋め込みができて便利だが、そのテクスチャの解像度やテクスチャ圧縮フォーマットを指定することができないのが難点。
Tripo などで生成した .glb をそのまま使うと、2048x2048 の raw RGB テクスチャがディスクとメモリを消費することになってしまう。

対処法としては、一旦 .gltf でエクスポートするというのが手軽。右クリックメニューから簡単にできる。ただし、テクスチャがデフォルトインポート設定になってしまうので、法線マップなどでは sRGB Color スイッチを適宜 off にしていく作業などが必要。

## glTF における法線マップの形式

Norm ではなく UNorm でエンコードされている。そのため、Unity でインポートする際は Normal 形式にするのではなく、カラーテクスチャとしてインポートしつつ sRGB Color スイッチを外すという作業が必要とされる。

# 両手分割キーボード

## 両手分割キーボードを使ってみた感想

40% キーボードを使ってみたが、非常にコンパクトで使い勝手が良い。

肩を開いた状態でキータイプができるのもかなり良い。エルゴノミックな利点があることは理解できたし、実感もできた。

Column staggered 配列のキーボードを選択したが、これが想像以上に異なる使用感を持つものであることを知った。通常の row staggered に慣れている状態からこれに移行することはかなり難しい。大きな移行コストが発生する。

しかしキーボードは毎日欠かさず使うものであるため、耐えれば自然と習得が進むという側面もある。

## Corne v4 両手分割キーボード

### macOS 側の Caps Lock 遅延

Quantum 機能を使って Shift キーのタップ操作に Caps Lock を組み込むということを行ったが、入力が無効化される現象が多発した。どうやら macOS 側の Caps Lock 遅延の仕様と QMK 側のタップ判定が競合してしまっているようだ。

これを避けるための手段の一つとして QMK Settings の Retro Tapping を有効化するというものがある。これによって、他のキーを同時に叩かなかった場合はタップ判定になるという挙動に変化する。個人的には Quantum はモディファイアにしか使っていないので、これで事足りる。

### Tap Hold Caps Delay

その後、実はこの現象に対してのワークアラウンドが QMK 側に用意されていることに気づいた。"Tap Hold Caps Delay" である。これがデフォルトの 80ms では短過ぎるらしい。100ms まで延ばしたところ、調子が良くなった。

Caps の設定については上記のように右往左往したが、最終的には Quantum の使用をやめて、単純にキーへの割り当てを行った。Caps Lock と Shift キーは連続して押下することが多く、Quantum が思ったように機能しないことが分かったため。

# バージョン管理ソフト

## jj (Jujutsu)

git との互換性を持つ新しいバージョン管理ソフト。直行性の高い比較的シンプルなコマンド体系を持っており、何かを行うのに複数のアプローチがあったりする git と比較して、 ストレートで覚えやすく使いやすいという利点がある。

しかし最大の利点は git との高い互換性にあるだろう。既存の git リポジトリを jj で管理することができ、管理側は追加のコストを払う必要が全くない。ただ単にローカルマシン側において jj を使いたい人は使えば良い、と言うだけ。今後も表向きは git で管理しつつ、ローカルではコマンドとして jj を使用する、と言うケースが増えていくのだろうと思う。

### jj の習い方

jj は比較的新しいコマンドである上に、最近も仕様の変更がかなりあったため、AI の言うことが当てにならない。古い情報に基づいた答えを返すことが多い。

個人的には NotebookLM を使用するのが良いと考えている。jj のソースリポジトリに含まれている `docs` 下の `.md` ファイルを全て NotebookLM に放り込めば、jj の事を何でも正しく答えてくれる AI の完成だ。これで git で行なっていたワークフローが jj ではどうなるか、と言うことについて逐次聞いていけばよい。

# AI Coding Agent

## ローカライゼーション

翻訳作業は LLM の本領であり、構造化されたテキストデータであれば非常に高いクオリティで翻訳ができる。しかし、適切な構造化が行われていないランダムなテキスト、例えば Web ページの生 HTML そのものなどを扱うのは問題がある。そういったデータを雑に丸投げすると、翻訳漏れやゴミの混入が発生してしまう。

aia-demo-kiosk では AI Assistant のチャットログ (Markdown) の翻訳を Claude Code に任せたが、翻訳を除外するケースの扱いに苦労し、結局翻訳を完遂することができなかった。これについては、翻訳工程に入る前の分離された工程として除外処理を行うなど、「あとは単純に翻訳だけすればよい状態」にまで整理しておくことによって、問題を回避することができた。

ゲームのローカライゼーションなども同様になるだろう。ローカライズを考えずに完成させたタイトルをいきなり翻訳せよと丸投げしたら、同じ問題が発生するであろうことが容易に想像できる。まずは既存のローカライゼーションのフレームワークに乗せる所から始め、そのフレームワークの上で自動化された翻訳処理を行う、というような流れにする必要があるだろう。

# Google Stitch

AI アシスト付きのデザインツール、ということになるだろうか。同様のツールに Claude Design が存在するが、Claude Design はスライドや印刷物など様々なデザインに応用できる汎用性を持っており、Google Stitch の方はウェブとアプリのデザインに特化したもの、という理解で良いと思う。

汎用性は Claude Design の方が高いのだろうけども、Webサイトを手っ取り早く作りたい、という需要の場合は、Stitch の方が格段にお手軽に使える。それに合わせたワークフローが設計されているため。

実際に aia-demo-kiosk の制作で使用してみたが、非常にお手軽だった。ただし、メンタルモデルがまだよく分からない……というか、例えば、ソースコードのどこまでを AI エージェントがコントロールを握っており、どこからが手動で編集するべき領域なのか、よく分かっていない。全てが AI に任せられれば良いのだろうけども、そういうわけでも無さそうだ。
